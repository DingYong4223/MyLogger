# Chrome 日志查看器能力规划

本文整理 MyLogger Chrome 插件后续可以扩展的日志相关能力，供后续排期和实现拆分使用。

## 当前基础

Chrome 插件当前已经具备以下基础能力：

- 打开本地日志文件，支持大文件主窗口虚拟渲染。
- 按关键字或正则过滤日志。
- 搜索当前可见日志并跳转上一个/下一个匹配。
- 保存过滤结果、搜索结果和分析结果为 `.txt`。
- 标记日志行，查看全部标记行。
- 点击筛选结果行打开日志上下文窗口，查看目标行前后日志。
- 加载 Android Studio 插件导出的断点 JSON。
- 调用本地后台服务做断点筛查，并展示断点日志字符串和相关日志。

## P0：优先实现

### 1. Tag/Level 统计面板

目标：帮助用户快速判断当前日志的结构和异常分布。

建议能力：

- 统计当前过滤结果中的 `V/D/I/W/E/F` 数量。
- 统计 Top Tags。
- 点击某个 Level 或 Tag 自动过滤主窗口。
- 支持一键清除统计面板触发的过滤条件。

实现要点：

- 基于 `state.visibleRows` 做本地统计即可。
- UI 可以放在右侧日志表格上方或左侧后台分析面板下方。
- 统计结果要跟随过滤条件变化实时刷新。

### 2. 断点筛查结果联动主日志

目标：把后台返回的断点命中结果和主日志定位串起来。

建议能力：

- 点击断点筛查右侧某条相关日志，打开对应主日志上下文。
- 点击左侧断点日志字符串，自动在主窗口过滤该字符串。
- 在断点筛查结果中展示“第几条匹配 / 总匹配数”。
- 支持从相关日志跳转到主窗口同一行附近。

实现要点：

- 后台已返回 `lineNumber`，前端可用该行号定位 `state.rows`。
- 复用当前日志上下文窗口。
- 注意过滤后的行号和原始行号的映射。

### 3. 多关键字高亮规则

目标：减少大日志中反复手工搜索的成本。

建议能力：

- 支持配置多组关键字和颜色。
- 内置常见规则：`Exception`、`FATAL EXCEPTION`、`ANR`、`timeout`、`error`、`failed`。
- 支持规则启用/禁用。
- 支持导入/导出规则 JSON。

实现要点：

- 规则数据可先保存在 `localStorage`。
- 渲染时只对当前虚拟可见行应用高亮。
- 避免全量 DOM 高亮，防止重新引入卡顿。

### 4. 搜索结果窗口虚拟渲染

目标：搜索命中很多时仍保持流畅。

建议能力：

- 搜索结果弹窗复用主窗口虚拟渲染能力。
- Prev/Next 基于数据索引跳转，不依赖全量 DOM。
- 点击搜索结果行打开日志上下文。

实现要点：

- 当前搜索结果仍使用 `renderRowsInto` 全量渲染。
- 可抽象一个 `VirtualLogTable` 帮主窗口、上下文窗口、搜索窗口共用。

## P1：中期能力

### 5. 日志时间线

目标：从时间维度发现异常密集区域。

建议能力：

- 按时间桶统计日志数量。
- 分别展示 Error/Warn/Info 的密度。
- 点击时间段跳转到对应日志位置。
- 支持只看异常时间段。

实现要点：

- Android logcat 已解析 `time` 字段，可按秒或分钟分桶。
- 无时间字段的普通日志可降级为按行号分布。

### 6. 异常摘要

目标：自动提取崩溃、ANR、超时等关键问题。

建议能力：

- 自动识别 `FATAL EXCEPTION`、`ANR`、`Caused by`、`TimeoutException`。
- 聚合异常堆栈。
- 展示异常摘要列表。
- 点击摘要打开对应上下文。

实现要点：

- 先做规则识别，不依赖 AI。
- Java/Kotlin 堆栈可按缩进行和 `at xxx` 连续行提取。

### 7. 日志模式聚类

目标：把大量类似日志归并成模板，降低噪音。

示例：

```text
request id=123 cost=45ms
request id=456 cost=60ms
```

聚合为：

```text
request id=* cost=*ms
```

建议能力：

- 展示日志模板、出现次数、首次/末次出现行。
- 点击模板过滤相关日志。
- 支持按 Tag 或 Level 分组聚类。

实现要点：

- 第一版可以用简单规则替换数字、UUID、hash、耗时等动态值。
- 后续再考虑更复杂的 Drain/Spell 类日志模板算法。

### 8. 收藏和批注

目标：支持复盘和协作。

建议能力：

- 对关键日志行添加备注。
- 导出“标记行 + 备注 + 上下文”为分析报告。
- 支持保存和恢复批注。

实现要点：

- 可先使用 `localStorage` 绑定文件名和行号。
- 后续可导出独立 JSON，便于分享。

### 9. 工作区保存

目标：恢复一次排查现场。

建议能力：

- 保存当前日志文件名、过滤条件、搜索词、断点文件名、标记行、批注。
- 下次打开同名文件时提示恢复。
- 支持导入/导出工作区 JSON。

实现要点：

- 不保存原始日志内容，只保存状态和配置。
- 注意本地文件路径在 Chrome 中不可稳定获取，恢复时可以提示用户重新选择文件。

## P2：高级能力

### 10. 多日志文件合并查看

目标：支持同时分析 app 日志、系统日志、后端日志。

建议能力：

- 打开多个日志文件。
- 按时间排序合并。
- 增加来源列。
- 支持按来源过滤。

实现要点：

- 需要统一时间解析和排序策略。
- 无时间字段时保留文件内原始顺序。

### 11. 实时 Tail

目标：查看正在增长的日志。

建议能力：

- 后端监控日志文件追加内容。
- 前端通过 WebSocket 或 SSE 接收增量。
- 支持暂停自动滚动。
- 支持“只看新日志”。

实现要点：

- Chrome 前端不能直接 tail 本地文件，需要本地后台服务配合。
- 后端可以先提供 `/tail` SSE 接口。

### 12. 类 SQL 查询

目标：比简单过滤更灵活地查询结构化日志。

示例：

```text
level:E tag:Network msg:timeout
```

或：

```sql
level = "E" and tag contains "Network" and message contains "timeout"
```

实现要点：

- 第一版建议做简单 DSL，不直接引入完整 SQL。
- 与当前 Filter/Search UI 并行存在。

### 13. AI 辅助分析

目标：对选中日志上下文生成问题摘要和排查建议。

建议能力：

- 选中一段日志后点击“分析这段日志”。
- 先本地裁剪和脱敏。
- 后端调用 LLM 生成摘要、关键证据和下一步建议。

实现要点：

- 不直接发送整份日志。
- 输出必须引用具体行号和日志片段。

### 14. 频率异常检测

目标：发现某类日志突然增多、减少或消失。

建议能力：

- 按时间窗口统计关键模板或 Tag 出现频率。
- 标记异常增长时间段。
- 点击异常点跳转上下文。

实现要点：

- 依赖日志时间字段。
- 可以先做阈值规则，再做更复杂的统计模型。

## 推荐路线

建议按以下顺序推进：

1. Tag/Level 统计面板。
2. 断点筛查结果联动主日志。
3. 多关键字高亮规则。
4. 搜索结果窗口虚拟渲染。
5. 异常摘要。
6. 日志模式聚类。
7. 工作区保存。
8. 实时 Tail 和 AI 辅助分析。

这条路线可以优先强化当前 Chrome 插件已经具备的交互基础，同时避免过早引入过重的后端或 AI 依赖。
