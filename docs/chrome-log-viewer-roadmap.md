# Chrome 日志查看器后续能力方案

本文整理 MyLogger Chrome 插件后续可以扩展的日志相关能力，供后续排期、拆分任务和评估实现成本使用。

## 当前基础

Chrome 插件当前已经具备以下能力：

- 通过“文件”菜单打开本地日志、常见文本文件和文件夹。
- 多文件多标签查看，过滤、搜索、筛选、标记和保存只作用于当前可见标签。
- 大日志主窗口虚拟渲染，内容默认不换行，支持横向滚动，行号列固定。
- 自动解析时间、级别、Tag 和内容；缺失列时降级展示原始内容。
- 支持多行 OR 过滤、正则过滤、大小写敏感过滤。
- 支持时间、级别、Tag 筛选。
- 支持搜索结果窗口、筛查结果窗口和日志上下文窗口。
- 支持行标记、全部标记、标记上下跳转和 Jump 行号跳转。
- 支持过滤结果、搜索结果和筛查结果保存为 `.txt`。
- 支持常用过滤新增、编辑、删除、拖拽排序、导入和导出。
- 支持最近 20 条历史过滤记录。
- 支持用户设置、配置备份和恢复。
- 支持打开 Android Studio 插件导出的断点 JSON，并通过本地 MyLogger 服务提取断点日志过滤规则。
- 支持两个已打开日志的左右分屏比较；左右两侧可独立过滤、标记、Jump 和复制。
- 支持随插件打包的实用工具页。

## P0：优先实现

### 1. 日志统计面板

目标：帮助用户打开日志后快速判断异常分布和主要日志来源。

建议能力：

- 统计当前可见结果中的 `V/D/I/W/E/F` 数量。
- 统计 Top Tags。
- 统计总行数、命中行数、无时间/无级别/无 Tag 行数。
- 点击某个 Level 或 Tag 后直接应用过滤。
- 支持一键清除统计面板触发的过滤条件。

实现要点：

- 基于当前 tab 的 `state.visibleRows` 做本地统计。
- 统计结果需要跟随顶部 Filter、时间、级别、Tag 筛选实时刷新。
- UI 可放在左侧工具栏或作为主表格上方的可收起统计条。
- 点击统计项时优先复用现有级别筛选和 Tag 筛选状态，避免引入第二套过滤状态。

### 2. 异常摘要

目标：自动提取崩溃、ANR、超时等关键问题，减少手动搜索成本。

建议能力：

- 自动识别 `FATAL EXCEPTION`、`ANR`、`Caused by`、`TimeoutException`、`NullPointerException`、`OutOfMemoryError`。
- 聚合连续 Java/Kotlin 堆栈。
- 展示异常摘要列表：类型、首行、摘要、相关行数。
- 点击摘要打开日志上下文窗口。
- 支持把异常摘要导出为文本报告。

实现要点：

- 第一版只做规则识别，不依赖 AI。
- 堆栈可以按 `at `、`Caused by:`、缩进行、空行终止等规则聚合。
- 摘要列表可以基于当前 tab 的全量 `rows` 生成，也可以支持“仅基于当前过滤结果”。

### 3. 多关键字高亮规则

目标：让用户在扫大日志时快速看到重点词。

建议能力：

- 支持配置多组关键字和颜色。
- 内置常见规则：`Exception`、`FATAL EXCEPTION`、`ANR`、`timeout`、`error`、`failed`、`crash`。
- 支持普通关键字和正则规则。
- 支持规则启用/禁用。
- 支持导入/导出高亮规则 JSON。

实现要点：

- 规则数据保存在 `localStorage`，后续可纳入设置备份。
- 渲染时只对虚拟可见行应用高亮，避免全量 DOM 处理。
- 需要定义高亮优先级，避免和当前搜索/过滤高亮互相覆盖。

### 4. 日志比较增强

目标：让比较窗口从“并排查看”升级为“差异定位”。

建议能力：

- 左右同步滚动开关。
- 按行号同步或按时间同步两种模式。
- 差异行高亮。
- 只看左侧独有、右侧独有、两侧都有的日志模板。
- 比较窗口内支持导出差异结果。

实现要点：

- 同步滚动可以先按滚动比例实现，再补时间同步。
- 差异识别第一版可基于归一化文本模板：替换数字、UUID、耗时、hash 等动态值。
- 比较窗口已有左右独立状态，可在每侧 state 外增加 compare session 层状态。

### 5. 工作区保存

目标：恢复一次排查现场。

建议能力：

- 保存打开文件名、文件大小、最后修改时间。
- 保存当前 tab、过滤词、搜索词、级别/Tag/时间筛选、标记行。
- 保存比较窗口选择和左右侧过滤状态。
- 保存断点文件名、常用过滤、历史过滤引用。
- 下次打开同名文件时提示恢复。
- 支持导入/导出工作区 JSON。

实现要点：

- 不保存原始日志内容，只保存状态和配置。
- Chrome 无法稳定获得本地绝对路径，恢复时使用文件名、大小、修改时间做匹配提示。
- 工作区 JSON 应和设置备份区分，避免混淆全局配置和单次排查状态。

## P1：中期能力

### 6. 日志时间线

目标：从时间维度发现异常密集区。

建议能力：

- 按秒或分钟统计日志数量。
- 分别展示 Error、Warn、Info 的密度。
- 点击时间段跳转到对应日志位置。
- 支持只看异常密集时间段。

实现要点：

- Android logcat 已解析 `timeValue`，可直接分桶。
- 没有时间字段的普通日志可降级为按行号分布。
- UI 可以做成主窗口底部小时间轴，避免遮挡表格。

### 7. 日志模式聚类

目标：把大量相似日志归并成模板，降低噪音。

示例：

```text
request id=123 cost=45ms
request id=456 cost=60ms
```

聚合为：

```text
request id=* cost=*ms
```

建议能力：

- 展示日志模板、出现次数、首次出现行、末次出现行。
- 点击模板过滤相关日志。
- 支持按 Tag、Level 分组聚类。
- 支持导出模板统计。

实现要点：

- 第一版用简单归一化规则替换数字、UUID、hash、时间戳、耗时、地址等动态值。
- 后续再考虑 Drain/Spell 类日志模板算法。

### 8. 批注和分析报告

目标：支持复盘和协作。

建议能力：

- 对关键日志行添加备注。
- 标记行支持分类或标签。
- 导出“标记行 + 备注 + 前后上下文”为分析报告。
- 支持保存和恢复批注。

实现要点：

- 第一版可用 `localStorage` 绑定文件指纹和行号。
- 导出报告建议使用 Markdown 或纯文本。
- 与工作区保存联动，便于恢复排查现场。

### 9. 多日志合并查看

目标：同时分析 app 日志、系统日志、后端日志。

建议能力：

- 选择多个已打开 tab 合并为一个新视图。
- 按时间排序合并。
- 增加来源文件列。
- 支持按来源过滤。
- 支持从合并视图跳回原文件上下文。

实现要点：

- 需要统一时间解析和排序策略。
- 无时间字段时保留文件内原始顺序，并在合并排序中降级处理。
- 现有多 tab 和比较窗口可以作为数据入口。

### 10. 断点筛查结果联动主日志

目标：把后台返回的断点命中结果和主日志定位串起来。

建议能力：

- 点击筛查结果右侧某条相关日志，打开对应主日志上下文。
- 点击左侧断点日志字符串，自动填入顶部 Filter。
- 在筛查结果中展示每条规则命中数量。
- 支持从筛查结果跳转到主窗口同一行附近。

实现要点：

- 后台结果中已有匹配日志和行号信息时，前端可用 `sourceLine` 定位。
- 需要注意过滤后的行号和原始行号映射。
- 可复用日志上下文窗口。

## P2：高级能力

### 11. 实时 Tail

目标：查看正在增长的日志文件。

建议能力：

- 本地 MyLogger 服务监听日志文件追加内容。
- 前端通过 WebSocket 或 SSE 接收增量。
- 支持暂停自动滚动。
- 支持只看新日志。
- 支持实时高亮和实时异常摘要。

实现要点：

- Chrome 前端不能直接 tail 本地文件，需要本地服务配合。
- 后端可先提供 `/tail` SSE 接口。
- 前端需要设计增量解析和虚拟列表追加策略。

### 12. 查询 DSL

目标：提供比普通 Filter 更强的结构化查询能力。

示例：

```text
level:E tag:Network msg:timeout
```

或：

```text
level = "E" and tag contains "Network" and message contains "timeout"
```

建议能力：

- 支持 `level`、`tag`、`time`、`message`、`line` 等字段查询。
- 支持 `AND`、`OR`、`NOT`。
- 支持保存 DSL 为常用过滤。

实现要点：

- 第一版建议做简单 DSL，不引入完整 SQL。
- DSL 与当前 Filter/Search UI 并行存在，避免破坏现有轻量过滤体验。

### 13. AI 辅助分析

目标：对用户选中的日志上下文生成摘要和排查建议。

建议能力：

- 选中一段日志或某个上下文窗口后点击“分析这段日志”。
- 输出问题摘要、关键证据、可疑原因、下一步建议。
- 输出必须引用具体行号。
- 支持脱敏规则预览。

实现要点：

- 不直接发送整份日志。
- 默认只发送用户明确选中的片段。
- 建议通过本地 MyLogger 后台服务调用 LLM，前端负责裁剪、预览和展示。

### 14. 频率异常检测

目标：发现某类日志突然增多、减少或消失。

建议能力：

- 按时间窗口统计关键模板、Tag 或 Level 的出现频率。
- 标记异常增长/下降时间段。
- 点击异常点跳转上下文。
- 可结合日志时间线展示。

实现要点：

- 依赖日志时间字段。
- 第一版可用固定阈值或移动平均。
- 后续可结合模板聚类做更准确的异常检测。

## 权限和上架风险建议

优先选择不新增 Chrome 权限的能力：

- 日志统计面板。
- 异常摘要。
- 多关键字高亮规则。
- 比较窗口增强。
- 工作区保存。
- 批注和报告。

需要谨慎评估的能力：

- 实时 Tail：需要本地服务能力，但不一定需要新增 Chrome 权限。
- AI 辅助分析：涉及日志片段传输和隐私说明，需要明确用户确认和脱敏策略。
- 浏览器网络日志、DevTools 面板、页面内容采集：可能需要新增权限或改变单一用途描述，建议暂缓。

## 建议实施顺序

1. 日志统计面板。
2. 异常摘要。
3. 多关键字高亮规则。
4. 日志比较增强：同步滚动、差异高亮。
5. 工作区保存。
6. 日志时间线。
7. 日志模板聚类。
8. 批注和分析报告。
9. 多日志合并查看。
10. 断点筛查结果联动。
11. 实时 Tail。
12. 查询 DSL。
13. AI 辅助分析。
14. 频率异常检测。
